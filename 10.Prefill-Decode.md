# PD 分离（Prefill-Decode Disaggregation）

> **架构流程**：`Ingress → vLLM Router → Prefill (kv_producer) / Decode (kv_consumer) → LMCache Server`
>
> PD 分离将 LLM 推理的 **Prefill（预填充）** 和 **Decode（逐 token 生成）** 阶段拆分到不同的 GPU 实例上运行，通过 LMCache Server 共享 KV Cache 实现跨阶段数据传输。

---

## 1. 为什么需要 PD 分离？

### 1.1 Prefill 与 Decode 的计算特性差异

| 阶段 | 计算类型 | GPU 利用模式 | 瓶颈 |
|------|---------|-------------|------|
| **Prefill** | 矩阵乘法密集型（Compute-bound） | 高 GPU 利用率，适合大 batch | 算力（FLOPS） |
| **Decode** | 内存带宽密集型（Memory-bound） | 低 GPU 利用率，每次只生成 1 token | 显存带宽 |

单实例混合部署时，Prefill 和 Decode 互相抢占 GPU 资源，导致：
- Prefill 大 batch 延迟 Decode 响应 → TTFT（首 token 延迟）抖动
- Decode 占用显存 → Prefill 可用 batch size 受限
- 无法对两种负载独立扩缩容

### 1.2 PD 分离后的收益


```
┌─────────────┐     ┌──────────────┐     ┌─────────────────┐
│   Router    │────▶│  Prefill Pod │────▶│  LMCache Server │
│ (prefix     │     │ (kv_producer)│     │  (KV Cache 池)   │
│  aware)     │     └──────────────┘     └────────┬────────┘
│             │                                      │
│             │     ┌──────────────┐                │
│             │────▶│  Decode Pod  │◀───────────────┘
└─────────────┘     │ (kv_consumer)│
                    └──────────────┘
```


- **独立扩缩容**：Prefill 按输入吞吐量扩缩，Decode 按并发连接数扩缩
- **TTFT 优化**：Prefill 实例可配置更激进的 `--max-num-batched-tokens`，大幅降低 TTFT
- **显存隔离**：Decode 实例无需为 Prefill 预留大 batch 显存，`--gpu-memory-utilization` 可更激进
- **硬件异构**：Prefill 可用高算力 GPU（H100/A100），Decode 可用低成本 GPU（L40S/RTX）

## 3. 组件说明

### 3.1 vLLM Router（PD 感知调度）

Router 是 PD 分离的调度中枢，基于请求特征自动分发：

- **新会话（无缓存前缀）** → 路由到 **Prefill** 实例（处理 input tokens + 产出 KV Cache）
- **续接会话（有缓存前缀）** → 路由到 **Decode** 实例（直接从 LMCache 加载 KV Cache 续写）
- **混合请求** → Router 根据前缀命中情况动态选择后端

核心参数：


```bash
--routing-logic prefixaware       # 前缀感知路由
--pd-separation enabled            # 启用 PD 分离
--prefill-label app=vllm-prefill  # Prefill Pod 标签选择器
--decode-label app=vllm-decode    # Decode Pod 标签选择器
```


### 3.2 Prefill 实例（kv_producer）

承担所有新请求的 **提示词处理** + **首 token 生成** + **KV Cache 产出**：


```bash
--kv-transfer-config '{"kv_connector":"LMCacheConnectorV1","kv_role":"kv_producer"}'
--max-num-batched-tokens 32768    # 大 batch 并行编码，降低 TTFT
--max-num-seqs 8                  # 限制并发序列数，让位给 batch
--enable-chunked-prefill          # 分块预填充，避免长文本阻塞
--gpu-memory-utilization 0.85     # 保留 15% 给 KV Cache 传输缓冲
```


**扩缩容信号**：
- 主触发器：`vllm:num_requests_waiting`（等待队列深度）
- 辅助触发器：`vllm:time_to_first_token_seconds` P99（TTFT 延迟兜底）

### 3.3 Decode 实例（kv_consumer）

承担续接请求的 **逐 token 生成**，从 LMCache 拉取已缓存的 KV Cache：


```bash
--kv-transfer-config '{"kv_connector":"LMCacheConnectorV1","kv_role":"kv_consumer"}'
--max-num-seqs 128                # 高并发，同时服务大量解码请求
--max-num-batched-tokens 8192     # 降低 batch tokens，Decode 不需要大 batch
--gpu-memory-utilization 0.95     # 激进用满显存（无需预留 Prefill 缓冲）
--enable-prefix-caching           # 启用前缀缓存命中
```


**扩缩容信号**：
- 主触发器：`vllm:num_requests_running`（当前运行请求数 / GPU 利用率）
- 辅助触发器：`vllm:request_generation_tokens_per_sec` P99（ITL 延迟兜底）

### 3.4 LMCache Server（KV Cache 共享池）

Prefill 产出的 KV Cache 写入 LMCache，Decode 从中读取。详见 [7.LMCache.md](7.LMCache.md)。

### 4.4 PD Router Deployment

> Router 使用 PD 感知调度，自动区分 Prefill 和 Decode 后端。


03-pd-router.yaml
```yaml
# 03-pd-router.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vllm-router
  namespace: llm-inference
spec:
  replicas: 1
  selector:
    matchLabels:
      app: vllm-router
  template:
    metadata:
      labels:
        app: vllm-router
    spec:
      serviceAccountName: vllm-router
      containers:
      - name: router
        image: lmcache/lmstack-router:latest
        args:
          - --host
          - 0.0.0.0
          - --port
          - "8000"
          - --service-discovery
          - k8s
          - --k8s-namespace
          - llm-inference
          - --k8s-service-discovery-type
          - pod-ip
          # ========== PD 分离：为 Prefill / Decode 使用不同标签 ==========
          - --k8s-label-selector
          - app in (vllm-prefill, vllm-decode)
          - --routing-logic
          - prefixaware
          # ========== PD 分离模式 ==========
          - --pd-separation
          - enabled
          - --prefill-label
          - app=vllm-prefill
```

## 7. 注意事项

1. **LMCache Server 是单点**：生产环境建议部署多副本 + 反亲和，或接入 Redis/Mooncake 作为二级缓存池
2. **Prefill/Decode 比例**：通常 1:2 ~ 1:4（取决于输入/输出 token 比），可通过 KEDA 独立调整 `maxReplicaCount`
3. **冷启动延迟**：全新技术栈首次请求需要经过 Router → Prefill → LMCache → Decode 全链路，TTFT 可能略高于混合部署
4. **vLLM 版本**：确保 Prefill 和 Decode 使用相同的 vLLM 版本和 LMCache 版本，否则 KV Cache 格式不兼容
5. **网络延迟**：LMCache 推荐部署在与推理 Pod 相同的节点或低延迟网络上，跨节点 KV Cache 传输会增加 TTFT 约 5-15ms
6. **GPU 显存规划**：
   - Prefill 实例：模型权重 (~2GB) + KV Cache 缓冲 (~10GB) = 至少 12GB
   - Decode 实例：模型权重 (~2GB) + KV Cache 工作区 (~8GB) = 至少 10GB
   - 使用 TimeSlice（[6.TimeSlice.md](6.TimeSlice.md)）的单卡可同时运行 1 Prefill + 1 Decode
